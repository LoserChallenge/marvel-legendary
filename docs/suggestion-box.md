# Suggestion Box — process-improvement ideas

A **persistent, cross-expansion** log of ideas for doing the work **faster, more efficiently, or better** — anywhere in the project (build process, tooling, workflow, docs). Raw candidates, not committed work.

**This is NOT a per-expansion initiative.** It accumulates across all expansion builds and beyond. Capture ideas **whenever and wherever** they surface — mid-build, mid-review, in passing. Don't wait for a dedicated review to log one.

**Lifecycle — mark in place, don't delete:** entries stay in the log and get marked as they're handled — `open` → `investigating` → `resolved` / `dropped` (or `promoted` when an idea graduates to committed work in `docs/priorities.md`, with the outcome noted here). Resolved/dropped entries remain as a record; they accumulate. Triaging a too-long list is a future problem, not a now problem.

**Discipline:** correctness is always the priority; speed/efficiency ideas only graduate if they don't compromise it. Treat every entry as a **hypothesis to validate**, not a settled fix — speed claims are a known over-confidence trap.

**Entry format:** `### [date] Title` · value tag · the idea · effort-to-start · caveat · **status**.

**★ PRIMARY LEVER (the real #1) — orchestrate reads across the session topology — `[behavioral]`, the largest REAL opportunity.**
The biggest real category is **messages (70.9k)**, and it is dominated by READS. So Legendary doesn't burn faster because of a config tax — it burns faster because coordinator/worker builds are read-heavy. Token cost ≈ *what gets read, by which session, when, and where*. The lever is allocating reads deliberately along those four axes (Paul's framing — this is the focus, not the config trims):

- **WHICH session reads the source.** The worker (in the worktree) reads the source detail — inventory, spec, code. The coordinator holds pointers + decisions and verifies the worker's REPORTED evidence (gate output, commit SHAs, game-test matrices), NOT the source. Don't re-read in the coordinator what the worker already consumes. Route workers/subagents to an exact file **+ section**, never "read the tracker."
- **WHERE the big read happens — push it into an isolated subagent.** A subagent that reads a large doc / PDF / codebase and returns only the answer keeps the bulk OUT of the main session's window. This session did this right three times — `rules-oracle`, the token-audit agent, and `codebase-navigator` each absorbed a large read and returned a summary; the main context paid the answer, not the file. Make this the default for any read over a few hundred lines.
- **WHEN — cut cross-session re-reading.** Each new coordinator session rebuilds context by re-reading handoffs + trackers. Keep handoffs LEAN (decisions + pointers, not restated source); reference canonical docs by path rather than copying their content forward into every handoff. A doc consulted repeatedly → one targeted extract, not a full re-read per session.
- **HOW — targeted, never bulk.** Grep/offset to the specific lines. Never full-read `script.js` (~17–19k lines — one read dwarfs the entire fixed overhead). Never pipe large `git show <branch>:<file> | grep` blocks into context (route code search to `codebase-navigator`; read a branch doc's slice via a subagent or `git show … > scratch` then `Read offset`).
- **Match verification depth to stakes** (Paul's existing rule, and the single biggest recurring coordinator read). Re-read source only when stakes warrant — a base-bug fix's exact placement (B16) did; a green-gated low-risk accept does not. Otherwise trust the worker's reported evidence.

**This-session evidence — what to avoid vs. what worked:** AVOID — the coordinator read inventory lines 820–1010 (all 8 schemes + 30 out-of-scope Ambitions) when it needed one ~6-line entry; pulled the full ~200-line prior handoff when `grep "Finding C|carryover"` would do. WORKED — the three subagent dispatches that isolated big reads. **Verdict:** largest real lever, but behavioral — adopt the discipline, don't build machinery (the diligence reads ARE the error-catching value; the coordinator/worker split stays). Durable home: a short "read orchestration" note in the `cc-coordinate` skill, NOT CLAUDE.md (which the slim below is trying to shrink).
